# 4. 요구사항의 시스템화 타당성 분석

* 요구사항 문서를 만들었다고 바로 개발할 수 있는 것은 아니다.

# 기술적 타당성 검토의 흐름

## 1. 성능 및 용량 산정 적정성 검토

* 예상 사용량을 산정한다.
* 과거 유사 프로젝트의 사용량을 반영하여 필요하면 재조정한다.
* 요구사항 중 성능 관련 비기능 요구사항을 만족하는지 판단한다.
* 요구사항 미충족 시 다시 조정한다.

### 검토 내용

* 서버 성능
* DB 성능
* 네트워크 처리량
* 동시 접속자 수
* 데이터 처리량

---

## 2. 시스템 간 상호 운용성 검토

* 서로 다른 목적을 가진 두 개 이상의 시스템이 정보를 주고받고 서비스를 공유하면서 정상적으로 동작할 수 있는지 검토한다.

### 연동 대상 시스템 확인

* 결제 시스템
* 문자 시스템
* 보험 시스템
* 택배 시스템

### 데이터 교환 가능 여부 확인

* 요청 데이터 형식 확인
* 응답 데이터 형식 확인
* 데이터 이름 및 구조 매핑 필요 여부 확인

예시

우리 시스템:
USER

외부 시스템:
MEMBER

→ 데이터 매핑 필요

### 서비스 제공 방식 확인

우리 서비스:
실시간 요청 처리

외부 시스템:
하루 1회 데이터 제공

문제:
동작 방식 차이 발생

결과:
추가 처리 또는 요구사항 수정 필요

### 요구사항 수정 예시

기존 요구사항:
사용자는 주문 후 즉시 배송 상태를 확인할 수 있어야 한다.

수정 요구사항:
사용자는 배송 상태 정보를 하루 1회 갱신된 기준으로 확인할 수 있다.

또는

사용자는 매일 오전 9시에 갱신된 배송 상태 정보를 확인할 수 있다.

# 3. IT 시장 성숙도 및 트렌드 부합성 검토

## 시장 성숙도

* 사용하려는 기술이나 제품이 시장에서 충분히 검증되었는지 확인하는 것

### 확인 항목

1. 사용하는 기업 수
2. 사용하는 사용자 수
3. 기술 지원 여부
4. 시장 점유율

### 검토 목적

* 안정성 확인
* 유지보수 가능 여부 확인
* 미래 기술 방향 확인

# 4. 기술적 위험 분석 검토

* 요구사항 구현 시 사용할 기술이 문제를 일으킬 가능성과 영향도를 분석하는 과정

## 주요 위험 요소

### 1. 복잡성

* 기술 구조가 지나치게 복잡한지 검토
* 기존 시스템 아키텍처와 충돌하는지 확인

예시:

* Backend : Spring Boot
* Frontend : React
* Cloud : AWS EC2

문제:
특정 클라우드 전용 솔루션 도입

결과:

* 구조 불일치
* 배포 방식 변경 필요
* 운영 방식 변경 필요
* 개발자 교육 필요

판단:
복잡성 위험 높음

---

### 기술 구조 과복잡 예시

요구사항:
사용자 행동 데이터를 기반으로 실시간 맞춤 추천 제공

필요 기술:

* 데이터 수집
* 실시간 처리
* 대용량 저장소
* 추천 알고리즘
* AI 모델
* 서버 확장 구조

문제:

* 구조가 너무 복잡함
* 장애 원인 파악 어려움
* 운영 난이도 증가

판단:
복잡성 위험 높음

---

### 2. 검증 여부

* 실제 프로젝트에서 충분히 사용되어 안정성이 검증되었는지 확인

#### 조직 내 무경험 예시

요구사항:
쿠버네티스 환경 운영

문제:

* 조직 경험 없음
* 설정 오류 가능성
* 장애 대응 어려움
* 교육 비용 증가
* 일정 지연 가능성

판단:
검증 위험 높음

---

#### 외부 지원 불가능 예시

상황:
개인이 만든 오픈소스 라이브러리 사용

문제:

* 최근 업데이트 없음
* 문서 부족
* 문의처 없음
* 사용 사례 부족

결과:

* 오류 해결 어려움
* 보안 대응 어려움
* 유지보수 불안정

판단:
검증 위험 높음
