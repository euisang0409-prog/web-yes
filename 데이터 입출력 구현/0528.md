# 물리 데이터 저장소 설계 및 데이터 모델링 정리

## 논리 데이터 저장소 확인하기

### 엔티티(Entity), 속성(Attribute), 관계(Relationship)

* 엔티티 : 저장하고 관리해야 하는 대상
* 속성 : 엔티티가 가지는 정보
* 관계 : 엔티티 간 연결 관계

### 정규화(Normalization)

* 데이터 중복을 최소화하기 위해 테이블을 분리하는 과정
* 이상 현상 제거 목적

---

# 물리 데이터 저장소 설계하기

## 반정규화(Denormalization)

테이블이 너무 많이 분리되어 JOIN이 과도하게 발생하면 조회 성능이 저하될 수 있다.
이를 해결하기 위해 테이블을 합치거나 중복 데이터를 허용하는 작업이다.

### 중복 테이블 추가

* 집계 테이블 추가
* 상태 컬럼 전용 테이블 생성
* 특정 부분만 포함하는 테이블 추가

### 테이블 조합

* 항상 같이 사용되는 여러 테이블을 하나로 합침

### 테이블 분할

#### 수직 분할

* 컬럼 사용 빈도 차이가 큰 경우 분리
* 자주 사용하는 컬럼 / 그렇지 않은 컬럼 구분

#### 수평 분할

* 특정 범위별 사용 빈도 차이가 큰 경우 분리

### 테이블 제거

* 더 이상 사용하지 않는 테이블 삭제

### 컬럼 중복

* 다른 테이블 컬럼을 현재 테이블에도 저장

# 테이블 제약조건

## 삭제 시 처리 옵션

### CASCADE

부모 데이터 삭제 시 자식 데이터도 삭제

```sql
FOREIGN KEY (customer_id)
REFERENCES customer(customer_id)
ON DELETE CASCADE;
```

### RESTRICT

참조 중이면 삭제 불가

```sql
FOREIGN KEY (customer_id)
REFERENCES customer(customer_id)
ON DELETE RESTRICT;
```

### SET NULL

부모 삭제 시 FK 값을 NULL 처리

```sql
FOREIGN KEY (customer_id)
REFERENCES customer(customer_id)
ON DELETE SET NULL;
```

---

## 수정 시 처리 옵션

### ON UPDATE CASCADE

```sql
FOREIGN KEY (customer_id)
REFERENCES customer(customer_id)
ON UPDATE CASCADE;
```

### ON UPDATE RESTRICT

```sql
FOREIGN KEY (customer_id)
REFERENCES customer(customer_id)
ON UPDATE RESTRICT;
```

### ON UPDATE SET NULL

```sql
FOREIGN KEY (customer_id)
REFERENCES customer(customer_id)
ON UPDATE SET NULL;
```

# 인덱스(Index) 설계

## 인덱스

* 데이터를 빠르게 찾기 위한 검색용 구조

## 인덱스 적용 기준

### 분포도가 10~15% 이하인 경우 효과적

분포도 공식

```text
(1 / 도메인 개수) X 100
```

예시

```text
회원번호 10000개
분포도 = (1 / 10000) X 100 = 0.01%
```

### 부분 처리 목적

```text
주문상태
완료     90000
배송중    8000
취소      2000
```

취소 데이터만 자주 조회하면 인덱스 활용 가능

### 조회 조건에 자주 사용되는 컬럼

```sql
SELECT *
FROM orders
WHERE member_id = 10
AND order_date >= '2026-01-01';
```

---

## 단일 인덱스

```sql
CREATE INDEX idx_user
```

## 결합 인덱스

```sql
CREATE INDEX idx_member_id_order_date
```

* 컬럼 순서 중요

## 인덱스 설계 고려사항

### 액세스 경로 영향 가능

#### Full Table Scan

* 전체 테이블 검색

#### Index Scan

* 인덱스 검색

### 인덱스 과다 생성 주의

* 수정 성능 저하
* 오버헤드 발생

### 추가 저장공간 필요

### 넓은 범위 조회에는 비효율 가능

# 뷰(View) 설계

## 뷰(View)

* SELECT 결과를 가상 테이블처럼 사용하는 객체

## 뷰 사용 이유

* 복잡한 SQL 단순화
* 특정 컬럼만 제공
* 조건에 맞는 데이터만 제공

---

## 뷰 속성

### REPLACE

```sql
CREATE OR REPLACE VIEW view_name AS
SELECT ...
```

### FORCE (Oracle 전용)

* 기본 테이블 없어도 생성 가능

### NOFORCE (Oracle 전용)

* 기본 테이블 반드시 존재해야 함

### WITH CHECK OPTION

```sql
CREATE VIEW view_name AS
SELECT ...
WITH CHECK OPTION;
```

### WITH READ ONLY

```sql
CREATE VIEW view_name AS
SELECT ...
WITH READ ONLY;
```

# 파티션(Partition)

## 파티션

* 큰 테이블을 여러 영역으로 나누어 저장하는 기술

## 목적

* 조회 범위 감소
* 백업/복구 효율 향상

---

# 인덱스 vs 파티션

## 인덱스

* 빠른 탐색 목적
* 목차 개념

## 파티션

* 데이터 분산 관리 목적
* 저장 영역 분리 개념

---

# 범위 분할(Range Partition)

```sql
CREATE TABLE orders(
    order_id INT,
    order_date DATE
)
PARTITION BY RANGE(YEAR(order_date))(
    PARTITION p2024 VALUES LESS THAN(2025),
    PARTITION p2025 VALUES LESS THAN(2026),
    PARTITION p2026 VALUES LESS THAN(2027)
);
```

## 파티션 프루닝

* 필요한 파티션만 조회

---

# 해시 분할(Hash Partition)

```sql
CREATE TABLE users(
    user_id INT,
    name VARCHAR(50)
)
PARTITION BY HASH(user_id)
PARTITIONS 4;
```

## 특징

* 데이터 균등 분산
* ID 기반 조회에 적합
* 범위 조회에는 비효율

---

# 조합 분할(Composite Partition)

```sql
CREATE TABLE orders(
    order_id INT,
    customer_id INT,
    order_date DATE
)
PARTITION BY RANGE(YEAR(order_date))
SUBPARTITION BY HASH(customer_id)
SUBPARTITIONS 4(
    PARTITION p2024 VALUES LESS THAN(2025),
    PARTITION p2025 VALUES LESS THAN(2026),
    PARTITION p2026 VALUES LESS THAN(2027)
);
```

## 특징

* 범위 + 해시 조합
* 대용량 데이터에 효과적

# 클러스터 설계

## 클러스터

* 관련 데이터를 물리적으로 가까운 위치에 저장하는 방식

## 목적

* JOIN 성능 향상
* 블록 접근 최소화

---

# Oracle 클러스터 예시

## 클러스터 생성

```sql
CREATE CLUSTER customer_order_cluster(
    customer_id NUMBER
);
```

## 클러스터 인덱스 생성

```sql
CREATE INDEX idx_customer_order_cluster
ON CLUSTER customer_order_cluster;
```

## 클러스터 테이블 생성

```sql
CREATE TABLE customer(
    customer_id NUMBER PRIMARY KEY,
    customer_name VARCHAR2(50)
)
CLUSTER customer_order_cluster(customer_id);
```

```sql
CREATE TABLE orders(
    order_id NUMBER,
    customer_id NUMBER
)
CLUSTER customer_order_cluster(customer_id);
```

# 클러스터 적용 기준

## 적합한 경우

* 데이터량이 큰 경우
* JOIN 빈도가 높은 경우
* GROUP BY / ORDER BY 빈번한 경우

## 부적합한 경우

* 수정이 잦은 컬럼
* 작은 테이블

# 단일 테이블 클러스터링

하나의 테이블 내부 데이터를 특정 기준으로 모아 저장

예시

* 주문일 기준 저장

---

# 다중 테이블 클러스터링

여러 테이블 데이터를 같은 기준으로 묶어서 저장

예시

* customer + orders 를 customer_id 기준으로 저장
