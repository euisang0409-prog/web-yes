# 논리 데이터 모델링

## 논리 데이터 모델링
- 데이터베이스 개발 과정의 첫 단계
- 업무 분석을 통한 자료의 흐름을 분석하여 현재 사용중인 양식, 문서, 장표를 중심으로 자료항목을 추출하여 엔티티와 속성들의 관계를 구조적으로 정의하는 단계

## 특징
- 논리 데이터 모델링을 진행할 때 요구사항을 충분히 수집하지 않으면 요구사항 변경에 따른 많은 비용이 발생한다.
- 이해당사자들과 의사소통의 보조자료로서 E-R 모델을 활용한다.
- 논리적 모델은 H/W나 S/W에 독립적이다.

# 함수적 종속성

## 함수적 종속성
- 어떤 속성의 값이 정해지면 다른 속성의 값도 하나로 정해지는 관계

A -> B

- A가 B를 결정한다.
- A에 의해 B가 결정된다.
- B는 A에 의해 함수적으로 종속된다.

| 학번 | 이름 | 학과 |
|---|---|---|
| 2026001 | 김철수 | 컴퓨터공학 |
| 2026002 | 이영희 | 전자공학 |
| 2026003 | 박민수 | 경영학 |

- 학번은 학생마다 유일하다.
- 학번 -> 이름
- 학번 -> 학과

---

# 함수적 종속성 종류

## 1. 완전 함수 종속 (Full Functional Dependency)
- 기본키 전체가 있어야 결정되는 경우

| 학번 | 과목번호 | 성적 |
|---|---|---|
| 1001 | J01 | A |
| 1001 | P01 | B |

- 기본키 : (학번, 과목번호)
- 학번만으로는 성적을 알 수 없음
- 과목번호만으로도 성적을 알 수 없음
- 둘 다 있어야 성적을 알 수 있음

---

## 2. 부분 함수 종속 (Partial Dependency)
- 기본키의 일부만으로 결정되는 경우

| 학번 | 과목번호 | 학생이름 |
|---|---|---|
| 1001 | J01 | 김철수 |

- 기본키 : (학번, 과목번호)
- 학번 -> 학생이름

---

## 3. 이행 함수 종속 (Transitive Dependency)
- 중간 속성을 거쳐 결정되는 경우

| 학번 | 학과번호 | 학과명 |
|---|---|---|
| 1001 | D01 | 컴퓨터공학 |

- 학번 -> 학과번호
- 학과번호 -> 학과명
- 학번 -> 학과명

# 다치 종속

## 다치 종속 (Multi Valued Dependency)
- 하나의 속성이 여러 개의 값을 가지는데 그 값들이 다른 속성과 서로 독립적인 경우

A는 여러 개의 B를 가진다.

| 학생 | 취미 | 자격증 |
|---|---|---|
| 김철수 | 축구 | 정보처리기사 |
| 김철수 | 축구 | SQLD |
| 김철수 | 게임 | 정보처리기사 |
| 김철수 | 게임 | SQLD |

- 취미와 자격증은 서로 관련이 없음
- 하나의 테이블에 넣으면 불필요한 조합이 발생함

분리 예시

학생 - 취미  
학생 - 자격증

# 조인 종속

## 조인 종속 (Join Dependency)
- 하나의 테이블을 여러 개로 분리했다가 다시 조인해야만 원래 의미가 유지되는 관계

| 교수 | 과목 | 강의실 |
|---|---|---|
| 홍길동 | Java | 101호 |
| 홍길동 | DB | 102호 |
