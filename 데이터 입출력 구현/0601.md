# 1. 논리 데이터 저장소 확인하기

## 논리 데이터 모델링

모델링 절차

```text
개념 모델링 → 논리 모델링 → 물리 모델링
```

구성요소

* 엔티티(Entity)
* 속성(Attribute)
* 관계(Relationship)

## 함수적 종속성

A가 B를 결정하는 관계

종류

* 완전 함수 종속
* 부분 함수 종속
* 이행 함수 종속
* 다치 종속
* 조인 종속

## 정규화

함수적 종속성을 제거하여 이상현상을 방지하는 과정

### 제1정규형(1NF)

모든 속성은 반드시 하나의 값만 가진다.

### 제2정규형(2NF)

부분 함수 종속 제거

모든 일반 속성은 기본키 전체에 종속되어야 한다.

### 제3정규형(3NF)

이행 함수 종속 제거

기본키가 아닌 속성끼리는 종속 관계가 존재하면 안 된다.

### 제4정규형(4NF)

다치 종속 제거

관련 있는 속성끼리 별도 테이블로 분리한다.

예시

학생 취미 테이블

| 이름 | 취미 |
| -- | -- |
| 철수 | 축구 |
| 철수 | 독서 |
| 철수 | 등산 |

학생 자격증 테이블

| 이름 | 자격증    |
| -- | ------ |
| 철수 | 정보처리기사 |
| 철수 | SQLD   |

### 제5정규형(5NF)

조인 종속 제거

분해된 테이블을 JOIN 했을 때 원래 결과가 복원되어야 한다.

## 논리 데이터 모델 검증

* 필요한 데이터 누락 여부 확인
* 기존 시스템 구조 분석
* 사용자 요구사항 반영 여부 확인
* 데이터 흐름 확인
* 데이터 모델 적절성 검증
* 접근 권한 및 통제 검증

# 3. 물리 데이터 저장소 구성

## 제약조건

### ON DELETE

#### CASCADE

부모 데이터 삭제 시 자식 데이터도 삭제

#### RESTRICT

참조 중이면 삭제 불가

#### NULLIFY

참조 데이터를 NULL 처리

### ON UPDATE

#### CASCADE

부모 데이터 수정 시 자식 데이터도 수정

#### RESTRICT

참조 중이면 수정 불가

#### NULLIFY

참조 데이터를 NULL 처리

예시

```sql
CONSTRAINT fk_name
FOREIGN KEY(column_name)
REFERENCES table_name(column_name)
ON DELETE CASCADE
```

## 인덱스 설계

목적

검색 성능 향상

분포도 계산

```text
분포도 = (1 / 컬럼값 종류 수) × 100
```

적용 기준

* 분포도 10~15%
* WHERE 조건 사용
* PK, UNIQUE 자동 생성
* 수정이 적은 컬럼
* 자주 사용하는 컬럼 조합

## 뷰(View)

가상의 테이블

### REPLACE

기존 뷰 재생성

```sql
CREATE OR REPLACE VIEW
```

### FORCE

기본 테이블 없어도 생성

### NOFORCE

기본 테이블 있어야 생성

### WITH CHECK OPTION

조건을 만족하는 데이터만 INSERT, UPDATE 가능

### WITH READ ONLY

조회 전용

## 클러스터

관련 데이터를 물리적으로 가까이 저장

특징

* 범위 조회 성능 향상
* 반복 조인 성능 향상

주의사항

* INSERT 성능 저하
* UPDATE 성능 저하
* DELETE 성능 저하

## 파티션

대용량 테이블 분할 기술

### 범위 분할

사용자가 범위를 지정

예

* 주문
* 로그
* 게시글
* 이력 데이터

### 해시 분할

DBMS가 자동 분산

예

* 회원
* 사용자
* 고객 데이터

### 조합 분할

범위 + 해시 등 여러 방식 혼합

파티셔닝 절차

1. 파티션 종류 결정
2. 파티션 키 선정
3. 파티션 수 결정

## 디스크 구성 설계

목적

* 저장 공간 효율 향상
* 입출력 경합 최소화
* 성능 향상

설계 항목

* 테이블
* 제약조건
* 인덱스
* 뷰
* 클러스터
* 파티션
* 테이블스페이스

# Trigger

END;
