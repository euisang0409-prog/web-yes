# 데이터 모델링 정의

## 실체(Entity)
- 시스템에서 저장하고 관리해야 하는 실제 대상이나 개념
- 예시: 학생, 강사, 과정(과목), 수강신청, 출결

## 속성(Attribute)
- 실체(Entity)에 들어가는 특징 정보

예시

학생
- 학번
- 이름
- 전화번호
- 이메일

## 실체(Entity)가 되기 위한 조건
- 업무에서 중요해야 한다.
- 여러 개가 존재해야 한다.
- 속성을 가진다.
- 각 행을 구분할 수 있어야 한다.

예시
- 학번으로 학생을 식별 가능

---

# 관계(Relation)

- 엔티티와 엔티티가 어떤 연결을 가지는지 표현하는 것
- 관계는 보통 동사 형태를 사용한다.

## 관계 종류
- 1:1 관계
- 1:N 관계
- N:M 관계

N:M 관계는 관계형 데이터베이스에서 중간 테이블을 추가하여 1:N 관계로 해소한다.

---

# 데이터 모델링 목적

1. 연관 조직의 정보 요구에 대한 이해
- 회사나 사용자가 어떤 정보를 필요로 하는지 파악 가능

2. 사용자, 설계자, 개발자 간 효율적인 의사소통 제공
- 모델이나 그림으로 표현하여 해석 차이를 줄임

3. 데이터 체계 구축을 통한 고품질 소프트웨어와 유지보수 비용 감소
- 처음부터 체계적으로 설계하여 수정 비용 감소

4. 신규 또는 개선 시스템 개발의 기초 제공
- 데이터 모델은 시스템 설계도의 역할 수행

---

# 무결성(Integrity)

- 데이터가 정확하고 일관성 있으며 신뢰 가능한 상태로 유지되는 것

## 1. 개체 무결성(Entity Integrity)
- PK는 중복될 수 없다.
- PK는 NULL 값을 가질 수 없다.

## 2. 참조 무결성(Referential Integrity)
- FK는 존재하는 데이터만 참조해야 한다.

## 3. 도메인 무결성(Domain Integrity)
- 컬럼에는 허용된 값만 저장 가능

## 4. 사용자 정의 무결성(User Defined Integrity)
- 업무 규칙에 맞는 데이터만 저장해야 한다.

- 규칙과 관계를 정확하게 표현

## 물리 모델링

# 논리 데이터 모델링

- 자료(양식, 문서, 장표)를 중심으로 자료 항목을 추출하여
엔티티와 속성의 관계를 구조적으로 정의하는 단계

## 논리 데이터 모델링 특징

1. 요구사항 수집이 부족하면 변경 비용 증가
2. E-R 모델을 의사소통 도구로 활용
3. HW/SW에 독립적이다.

# 정규화(Normalization)

- 데이터 중복을 최소화하고 일관성을 보장하기 위한 개념

## 목적
- 데이터 중복 제거
- 관리 편의성 향상
- 저장 공간 최소화

---

# 정규화를 하지 않았을 때 문제점

## 1. 삽입 이상(Insertion Anomaly)

- 원하지 않는 데이터도 함께 입력해야 하는 문제

예시

| 강사명 | 강사전화 |
|---|---|
| 홍길동 | 010-3333 |

입력하려면 다른 데이터도 필요함

---

## 2. 갱신 이상(Update Anomaly)

- 하나의 정보를 수정할 때 여러 곳을 수정해야 함

| 수강생ID | 수강생명 | 과정명 | 강사명 | 강사전화 |
|---|---|---|---|---|
| 1001 | 김철수 | Java | 홍길동 | 010-9999 |
| 1002 | 이영희 | Java | 홍길동 | 010-1111 |

일부만 수정되면 데이터 불일치 발생

---

## 3. 삭제 이상(Delete Anomaly)

- 데이터를 삭제할 때 필요한 정보도 함께 삭제되는 문제

| 수강생ID | 수강생명 | 과정명 | 강사명 | 강사전화 |
|---|---|---|---|---|
| 1003 | 박민수 | Python | 김영수 | 010-2222 |

수강생 삭제 시 과정/강사 정보도 삭제됨

---

# 함수적 종속성(Function Dependency)

- 어떤 컬럼이 다른 컬럼의 값을 결정하는 관계

A → B
- A가 B를 결정한다.
- B는 A에 함수적으로 종속된다.

예시

| 학번 | 이름 | 학과 |
|---|---|---|
| 2025001 | 김철수 | 컴퓨터공학 |
| 2025002 | 이영희 | 경영학 |

학번 → 이름  
학번 → 학과

---

# 함수적 종속성 종류

## 1. 완전 함수 종속(Full Functional Dependency)

- 기본키 전체에 의존

예시

| 학번 | 과목코드 | 성적 |
|---|---|---|
| 2025001 | JAVA01 | A |

기본키
(학번, 과목코드)

(학번, 과목코드) → 성적

---

## 2. 부분 함수 종속(Partial Dependency)

- 기본키 일부에만 의존

예시

| 학번 | 과목코드 | 학생이름 |
|---|---|---|
| 2025001 | JAVA01 | 김철수 |

학번 → 학생이름

부분 함수 종속 발생

### 문제점
- 데이터 중복
- 이상현상 발생

---

## 3. 이행 함수 종속(Transitive Dependency)

A → B  
B → C

그러면 A → C 가능

예시

학번 → 학과명  
학과명 → 학과위치

---

## 4. 다치 종속(Multivalued Dependency)

- 하나의 값에 여러 값이 연결됨

예시

| 학생 | 취미 |
|---|---|
| 김철수 | 축구 |
| 김철수 | 게임 |

학생 →→ 취미

---

## 5. 조인 종속(Join Dependency)

- 분해한 테이블을 다시 조인했을 때 원본으로 복원되는 관계

---

# 정규형

# 제1정규형(1NF)
- 모든 속성은 하나의 값만 가져야 한다.

# 제2정규형(2NF)
- 부분 함수 종속 제거

## 예시

학생

| 학생ID(PK) | 학생명 |
|---|---|
| 1001 | 김철수 |

과목

| 과목ID(PK) | 과목명 |
|---|---|
| A01 | Java |

수강정보

| 등록번호(PK) | 학생ID(FK) | 과목ID(FK) |
|---|---|---|

---

# 제3정규형(3NF)

- 이행 함수 종속 제거

## 예시

학생

| 학번 | 이름 | 학과명 |
|---|---|---|

학과

| 학과명 | 학과전화 |
|---|---|

---

# 제4정규형(4NF)

- 다치 종속 제거
- 관계 없는 속성 분리

---

# 제5정규형(5NF)

- 조인 시 가짜 데이터가 발생하지 않도록 분해

---

# 정규화 장단점

## 장점
- 유연한 데이터 구축 가능
- 데이터 정확성 향상

## 단점
- 물리적 접근 복잡
- JOIN 증가

---

# 모델 작성 기법

- ERD를 이해하기 쉽게 배치하는 방법

## 규칙

1. 엔티티를 정렬하여 배치
2. 업무 흐름 순서대로 배치
3. 관계 엔티티는 가까이 배치
4. 관계선은 수직/수평 사용
5. 적절한 간격 유지

# 논리 모델링 검증

## 1. 필요한 데이터 파악 여부
- 입력/출력 데이터 확인
- 연계 데이터 확인
- 신규 데이터 확인

## 2. 현재 시스템 데이터 현황 분석 여부
- DB 구조
- 백업 현황
- 공통코드
- 외부 연계 데이터 확인

## 3. 사용자 요구사항 분석 여부
- 볼륨
- 분산 구조
- 제약조건
- 백업/복구 정책
- 데이터 전환
- 보안 요구사항 확인

## 4. 데이터 흐름 정의 여부
- 프로세스별 입출력 데이터 정의
- 공유 데이터 정의

## 5. DB 설계 기준 정의 여부
- 테이블/컬럼 작성 규칙
- 모델 설계 지침 확인

## 6. 데이터 모델링 적절성 확인
- 논리 모델 완전성
- 무결성 보장 여부
- 업무 규칙 일치 여부

## 7. 엔티티와 프로세스 관계 확인
- CRUD Matrix 활용

## 8. 접근권한 및 통제 분석 여부
- 데이터 중요도
- 암호화 가이드라인 확인

- 조회 성능 향상을 위해 일부러 중복 허용

## 반정규화 이유
- 데이터 증가 시 조회 성능 저하 방지

---

# 반정규화 종류

## 중복 테이블 추가

- 진행 테이블
- 특정 부분 테이블

## 테이블 조합
- 자주 함께 사용하는 테이블 통합

- 수직 분할
- 수평 분할

## 테이블 삭제

## 컬럼 중복
