# 반정규화 (Denormalization)

## 개념
- 정규화를 일부 완화
- SELECT 성능 향상을 위해
- 데이터 중복을 허용하는 설계 방법

## 반정규화를 하는 이유

- JOIN이 너무 많음
- 데이터 증가 시 조회 속도 저하
- 자주 조회하는 데이터를 빠르게 사용하기 위함

## 핵심 개념

- JOIN 감소
- 조회 성능 향상
- 계산 비용 감소
- 자주 사용하는 데이터 통합

---

# 반정규화 유형

## 1. 중복 테이블 추가

### 목적
- 조회 성능 향상
- JOIN 감소

### 예시

```sql
CREATE TABLE order_summary(
    order_id INT PRIMARY KEY,
    customer_name VARCHAR(50),
    product_name VARCHAR(100),
    price INT,
    order_date DATE
);
```

```sql
INSERT INTO order_summary
SELECT
    o.order_id,
    c.customer_name,
    p.product_name,
    p.price,
    o.order_date
FROM orders o
JOIN customers c
ON o.customer_id = c.customer_id
JOIN products p
ON o.product_id = p.product_id;
```

- 원본 변경 시 Trigger 사용 가능

---

## 2. 집계 테이블 추가

### 목적
- 합계 / 평균 / 개수 미리 저장

### 사용 예시
- 일별 매출
- 월별 주문 건수
- 평균 주문 금액

---

## 3. 진행 테이블 추가

### 목적
- 진행 상태 빠른 조회

### 예시
- 배송중
- 결제완료
- 주문접수

---

## 4. 특정 부분만 포함하는 테이블 추가

### 목적
- 자주 사용하는 컬럼만 분리
- 조회 성능 향상

# 테이블 분할

## 수직 분할

### 개념
- 컬럼 기준 분할

### 사용 상황
- 자주 사용하는 컬럼 분리

---

## 수평 분할

### 개념
- 행(Row) 기준 분할

### 예시
- 연도별 주문 테이블 분리

```sql
CREATE TABLE ORDERS_2024(
    order_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    price INT,
    order_date DATE
);
```

```sql
INSERT INTO ORDERS_2024
SELECT *
FROM orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31';
```

### 사용 상황
- 날짜별 조회
- 지역별 조회
- 상태별 조회

# 테이블 제거

## 목적
- 더 이상 사용하지 않는 테이블 삭제

### 예시

```sql
ALTER TABLE customer
ADD address VARCHAR(100);
```

```sql
UPDATE customer c
JOIN customer_address a
ON c.customer_id = a.customer_id
SET c.address = a.address;
```

- 실제 운영에서는 백업 후 삭제

# 컬럼 중복

## 개념
- 다른 테이블 컬럼을 현재 테이블에도 저장

## 장점
- JOIN 없이 조회 가능

## 단점
- 데이터 수정 시 중복 수정 필요

---

# 인덱스 (INDEX)

## 특징
- 별도의 정렬된 자료구조
- 검색 속도 향상

## 주의점

### 1. 수정 시 인덱스도 변경됨
- INSERT / UPDATE / DELETE 비용 증가

### 2. 인덱스 많으면 오버헤드 증가
- 저장 공간 추가 사용
- 변경 작업 느려짐

### 3. 넓은 범위 조회는 효과 낮음

```sql
SELECT *
FROM orders
WHERE order_date BETWEEN '2020-01-01' AND '2026-12-31';
```

- 조회 범위가 너무 크면
- Full Table Scan 선택 가능

---

# VIEW

## 개념
- SELECT문을 저장한 가상 테이블

## 장점
- 복잡한 SQL 단순화
- 특정 컬럼만 공개 가능
- 특정 조건 데이터만 제공 가능

---

# VIEW 속성

## REPLACE

```sql
CREATE OR REPLACE VIEW active_member_view AS
SELECT member_id, name, email
FROM member
WHERE status = 'ACTIVE';
```

---

## FORCE

```sql
CREATE FORCE VIEW member_view AS
SELECT member_id, name
FROM member;
```

- 테이블 없어도 생성 가능

---

## NOFORCE

```sql
CREATE NOFORCE VIEW member_view AS
SELECT member_id, name
FROM member;
```

- 기본 테이블 존재 시만 생성

---

## WITH CHECK OPTION

```sql
CREATE VIEW active_member_view AS
SELECT member_id, name, status
FROM member
WHERE status = 'ACTIVE'
WITH CHECK OPTION;
```

- 조건을 벗어나는 UPDATE 금지

---

## WITH READ ONLY

```sql
CREATE VIEW active_member_view AS
SELECT member_id, name, status
FROM member
WHERE status = 'ACTIVE'
WITH READ ONLY;
```

- 조회만 가능

---

# VIEW 설계 고려사항

## 1. VIEW는 실제 테이블을 조회한다
- 실제 데이터 저장 X

## 2. 복잡한 VIEW는 느릴 수 있다
- JOIN / 집계 많으면 성능 저하

## 3. WHERE 조건 인덱스 고려

```sql
CREATE VIEW active_member_view AS
SELECT member_id, name, email
FROM member
WHERE status = 'ACTIVE';
```

# 파티션 (Partition)

## 개념
- 큰 테이블을 여러 저장 영역으로 나누는 방법

## 목적
- 대용량 데이터 성능 향상
- 조회 / 백업 / 복구 효율 증가

---

# 파티션 종류

## 범위 분할 (Range Partition)

```sql
CREATE TABLE orders(
    order_id INT,
    order_date DATE
)
PARTITION BY RANGE(YEAR(order_date))
(
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026),
    PARTITION p2026 VALUES LESS THAN (2027)
);
```

## 특징
- 사용자는 일반 테이블처럼 사용
- 내부 저장 영역만 분리

## 사용 예시
- 주문 데이터
- 로그 데이터
- 게시글 데이터
- 이력 데이터

---

# 반정규화 vs 파티션 차이

| 구분 | 반정규화 | 파티션 |
|---|---|---|
| 개념 | 테이블 구조 변경 | 저장 영역 분리 |
| 목적 | JOIN 감소 | 대용량 처리 |
| 방식 | 테이블 통합/중복 | 내부 영역 분할 |
| 사용자 관점 | 실제 테이블 변경 | 일반 테이블처럼 사용 |
