# 실습 시나리오

A호텔은 객실 예약 업무를 엑셀과 전화 기록으로 관리하고 있습니다.

현재 다음 문제가 발생하고 있습니다.

- 고객 정보와 예약 정보가 여러 파일에 중복 저장됨
- 특정 객실의 예약 여부 확인이 어려움
- 고객별 예약 이력 조회가 어려움
- 객실별 현재 상태를 수작업으로 확인함
- 월별 예약 건수와 매출 통계를 수작업으로 계산함

호텔 관리자는 이를 개선하기 위해 **호텔 예약 관리 데이터베이스 시스템**을 구축하려고 합니다.

### 문제3. 다음 논리 모델 관계를 분석하여 **외래키를 추가할 테이블, 외래키명, 필요한 제약조건**을 작성하시오.

**제공 관계**

```
고객 1 : 예약 N
→ 예약은 반드시 고객에 연결되어야 한다.

객실 1 : 예약 N
→ 예약은 반드시 객실에 연결되어야 한다.

예약 1 : 결제 1
→ 결제는 반드시 예약에 연결되어야 한다.

예약 1 : 체크인 1
→ 체크인은 반드시 예약에 연결되어야 한다.
```

**외래키 칼럼명 규칙**

- 참조하는 테이블의 기본키 이름을 그대로 사용한다.
- 기본키는 `_id` 형식을 사용한다.

**제약조건 규칙**

- 관계 설명에 **"반드시"** 라는 조건이 있으면 `NOT NULL`을 추가한다.
- `1:1 관계`는 외래키에 `UNIQUE`를 추가한다.
- 관계 연결은 `FOREIGN KEY`를 사용한다.

| 관계 | FK 추가 테이블 | FK적용 컬럼 | 제약조건 |
| --- | --- | --- | --- |
| 고객-예약 |  |  |  |
| 객실-예약 |  |  |  |
| 예약-결제 |  |  |  |
| 예약-체크인 |  |  |  |

정답 : 
관계			FK추가 테이블	FK적용 컬럼	제약조건
고객-예약		reservation		customer_id	NOT NULL, foreign key
									REFERENCES customer(customer_id)
객실-예약		reservation		room_id		NOT NULL, FOREIGN KEY
									REFERENCES room(room_id)
예약-결제		payment		reservation_id	NOT NULL, UNIQUE, FOREIGN KEY
									REFERENCES reservation(reservation_id)
예약-체크인	check_in		reservation_id	NOT NULL, UNIQUE, FOREIGN KEY
									REFERENCES reservation(reservation_id)
