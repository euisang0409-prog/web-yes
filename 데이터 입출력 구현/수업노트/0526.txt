요구사항 확인 -> 데이터 입출력 구현

논리 데이터저장소 확인하기

데이터 모델링을 정의
실체(Entity) -> 테이블
- 시스템에서 저장하고 관리해야 하는 실제 대상이나 개념
- 학생, 강사, 과정(과목), 수강신청, 출결
속성(Attribyute)
- 실체(Entity)에 들어가는 특징 정보
예시
학생
- 학번
- 이름
- 전화번호
- 이메일

실체(Entity)가 되기 위한 조건
- 업무에서 중요해야 한다.
- 여러 개가 존재해야 한다(학생은 여러명이 존재한다.)
- 속성을 가진다.
- 각 행을 구분할 수 있어야 한다.(학번으로 학생을 식별할 수 있다.)

관계(Relation)
- 엔티티와 엔티티가 어떤 연결을 가지는지 표현하는 것
- 관계는 보통 동사 형태를 사용한다.
- 1:1 관계
- 1:N 관계
- N:M 관계 -> 관계형 데이터베이스에서 N:M 관계는 중간 테이블을
하나 더 만들어서 1:N 관계로 해소를 시켜줘야 한다.

데이터 모델링 목적
- 연관 조직의 정보요구에 대한 이해를 할 수 있다.

ex) 회사나 사용자가 어떤 정보를 필요로 하는지 파악할 수 있다.
- 사용자, 설계자, 개발자 간에 효울적인 의사소통 제공

ex) 같은 내용을 그림이나 모델로 표현하면 서로 다르게 해석하는 문제가 줄어든다.
- 데이터 체계 구축을 통한 고품질 S/W와 유지보수 비용의 감소효과

ex) 처음부터 데이터를 체계적으로 설계하면 나중에 수정할 일이 줄어든다.
- 신규 또는 개선 시스템의 개발 기초를 제공한다.

ex) 시스템 개발시 데이터 모델은 설계도 역할을 한다.

데이터 모델링을 하면 무결성을 보장할 수 있다.
무결성
- 데이터가 정확하고, 일관성 있고, 신뢰할 수 있는 상태로 유지되는 것

1. 개체 무결성
- 각 행을 구분할 수 있어야 한다
- 기본키는 중복이될 수 없으며, NULL값도 가질 수 없다.

2. 참조 무결성(FK)
- 다른 테이블을 참조할 때, 존재하는 데이터만 참조해야 한다.

3. 도메인 무결성(CHECK)
- 컬럼에 허용된 값만 저장해야 하는 원칙

4. 사용자 정의 무결성(CHECK)
- 업무 규칙에 맞는 데이터만 저장해야 한다는 원칙

데이터 모델링 절차

개념
- 주요 엔티티타입, 기본 속성, 관계, 주요 업무를 포함한다.

논리
- 개념모델로부터 업무영역의 업무 데이터 및 규칙을 구체적으로
표현한 모델
- 모든 업무 데이터를 정규화 하여 모델링한다.
- 규칙과 관계를 완전하고 정확하게 표현한다.

물리
- 시스템의 설계적 및 정보 요건을 정확하고 완전하게 표현한 모델
- 데이터베이스 생성을 위한 물리구조로 변환

논리 데이터 모델링
- 자료(사용중인 양식, 문서, 장표)를 중심으로 자료항목을 추출해서
엔티티와 속성들의 관계를 구조적으로 정의하는 단계

논리 데이터 모델링의 특성
1. 요구사항을 충분히 수집하지 않으면,  요구사항 변경에 따른 많은
비용이 발생하게 된다.

2. 모든 이해당사자들과 의사소통의 보조자료로서 E-R모델을 활용한다.

3. 논리적 모델은 HW나 SW에 독립적이다.

정규화(Normalization)
- 중복성을 최소화하고 정보의 일관성을 보장하기 위한 개념
- 데이터 중복 배제로 데이터 관리 편의성 제고 및 자료 저장 공간의 최소화

정규화를 하지 않았을 때 문제점
1. 삽입이상
- 새로운 데이터를 넣으려고 할 때 원하지 않는 데이터도 함께 넣어야 하는 문제
넣고싶은 데이터
강사명 : 홍길동
강사전화 : 010-3333
수강생ID	수강생명	과정명	강사명	강사전화

2. 갱신이상
- 하나의 정보를 수정할 때 여러 곳을 모두 수정해야 하는 문제
수강생ID	수강생명	과정명	강사명	강사전화
1001		김철수	Java		홍길동	010-9999
1002		이영희	Java		홍길동	010-1111
부분만 바뀌게 되면 어떤 것이 진짜인지 알 수 없다.

3. 삭제이상
- 데이터를 삭제했는데 원하지 않는 정보도 함께 삭제되는 문제

수강생ID	수강생명	과정명	강사명	강사전화
1003		박민수	Python	김영수	010-2222

박민수가 학원을 그만둠
과정과 강사정보까지 같이 삭제가 된다.
필요 없는 데이터를 삭제하다가 필요한 정보도 같이 삭제되는 문제

함수적 종속성
- 어떤 컬럼이 다른 컬럼의 값을 결정하는 관계
A -> B
A가 B를 결정한다
B는 A에 함수적으로 종속된다.
A값을 알면 B값을 정확히 알 수 있다.

학번      이름      학과
2025001   김철수   컴퓨터공학
2025002   이영희   경영학
2025003   박민수   디자인

학번 -> 이름
학번 -> 학과

함수적 종속성의 종류

1. 완전함수종속(Full Functional Dependency)
- 기본키 전체에 의존하는 경우

학번   	과목코드   성적
2025001   	JAVA01  	A
2025001   	DB01   	B

기본키
(학번, 과목코드)

함수적 종속
(학번, 과목코드) -> 성적

성적을 알려면
- 학번만으로는 부족
- 과목코드만으로도 부족
- 둘다 필요하기 때문에 완전 함수 종속

2. 부분 함수 종속(Partial Dependency)
- 기본키 일부만으로 결정되는 경우

학번		과목코드	학생이름
2025001	JAVA01	김철수

기본키
(학번, 과목코드)

종속 관계
학번 -> 학생 이름
- 학생 이름이 복합키 전체가 아니라 일부(학번)에만 의존한다.
- 부분함수 종속이 발생

부분 함수 종속의 문제점
- 데이터 중복과 이상현상을 발생시킨다.

학생ID   과목ID   학생명   과목명   담당교수
1001      A01      김철수   Java      홍길동
1001      A02      김철수   Python  김영수
1002      A01      이영희   Java      홍길동

이행 함수 종속(Transitive Dependency)
- A가 B를 결정하고, B가 C를 결정하는 경우
학번      	학과명      		학과위치
2025001     컴퓨터공학  	 A동

학번 -> 확과명
확과명 -> 학과위치
그러면 학번 -> 학과위치가 간접적으로 결정

4. 다치 종속(Multivalued Dependency)
- 다치 = 여러 개의 값
- 하나의 값에 여러 개의 값이 연결되는 것

학생      취미
김철수   축구
김철수   게임
김철수   영화감상

학생 ->-> 취미

5. 조인 종속(Join)
- 테이블을 여러 개로 나누었을 때, 다시 조인하면 원래 테이블로 정확히 복원되는 관계

학생 테이블
학번   이름      학과코드
1   	김철수   C01

학과 테이블
학과코드   학과명
C01      	컴퓨터공학과

두 테이블을 학과코드로 조인하면 다음처럼 합쳐진다.
학번		이름		학과코드		학과명
1		김철수	C01			컴퓨터공학과

제 1 정규화 (1NF)
- 모든 속성은 반드시 하나의 값을 가져야 한다.

제 2 정규화 (2NF)
- 부분함수 종속 제거하기

학생ID(PK) 		 과목ID(PK)   	학생명   	과목명
1001   		A01   		김철수   	Java
1001   		A02   		김철수   	Python

학생ID(PK)	학생명

과목ID(PK)	과목명

등록번호(PK)	학생ID(FK)	과목ID(FK)

제 3 정규형(3NF)
- 이행 함수 종속성을 제거

학번  이름     학과명
1      	김철수   컴퓨터공학과

학과명      학과전화
컴퓨터공학과   02-1111

학번 -> 학과명
학과명 -> 학과전화

제 4 정규화
- 다치종속 제거하기
- 관계가 없는 속성을 분리하기

제 5 정규화
- 테이블을 분해했을 때, 다시 조인하면서 가짜 데이터가 생기지 않도록 만드는 것

정규화를 진행하게 되면
- 유연한 데이터 구축이 가능
- 데이터의 정확성이 높아진다.

단점
- 물리적 접근이 복잡
- 길이가 짧은 데이터 생성으로 과도한 조인 발생

모델 작성 기법
- ERD를 그릴 때 엔티티와 관계를 보기 이해하기 쉽게 배치하는 방법
- 업무 흐름이 잘 보이도록 정리해서 배치하는 규칙

엔티티들은 정렬하여 배치한다.
- 무작위로 배치하지 않고 줄을 맞춰 정렬해서 배치

업무흐름의 진행 순서와 관련된 엔티티들은 진행순서를 고려하여
좌에서 우, 위에서 아래로 배치한다.


중심에 배치된 엔티티와 관계를 가진 엔티티를 가까이 배열한다.

관계는 사선이 아닌 수직, 수평선을 사용한다.

엔티티를 너무 붙히지 않고, 적절한 간격을 둔다.

논리 모델링 검증
- 데이터베이스를 실제로 만들기 전에, 분석한 데이터 구조가 업무에 맞게
잘 정리되었는지 확인하는 과정

1. 시스템을 만들기 위해 필요한 데이터를 파악했는지
- 입력데이터, 출력데이터가 파악됐는지
- 데이터가 다른 업무나 시스템에서도 사용하는지 파악
- 새롭게 필요한 데이터가 있는지 파악 

2. 현재 운영중인 시스템의 데이터 현황 분석이 이루어졌는지 확인
- 데이터베이스 구조, 분산, 백업현황이 정확히 파악되었는지
- 데이터 속성, 공통코드가 정확히 파악되었는지
- 외부 연계 데이터 속성이 정확히 파악되었는지

3. 데이터베이스에 대한 사용자 요구사항이 도출되고 분석되었는지
- 볼륨, 분산구조, 제약조건에 대한 요구사항이 도출되고 분석되었는지 확인
- 타 시스템과 연계 데이터에 대한 요구사항이 도출되고 분석되었는지 확인
- 데이터베이스 백업 및 복구 정책에 대한 요구사항이 도출되고 분석되었는지 확인
- 초기 데이터 구축방안에 대한 요구사항이 도축되고 분석되었는지 확인
- 기존 데이터의 전환방안에 대한 요구사항이 도출되고 분석되었는지 확인
- 무결성, 데이터 보안성 수준에 대한 요구사항이 도축되고 분석되었는지 확인

4. 데이터의 흐름이 명확하게 정의되었는지 화인
데이터의 흐름 : 데이터가 어디서 생성되고, 어디로 이동하며, 어디에서 사용되는지 나타내는 것

- 프로세스별 입출력 데이터가 명확히 정의되었는지 확인
- 프로세스 간 공유 및 연계 데이터가 명확히 정의되었는지 확인한다.
ㄴ 하나의 데이터는 여러 프로세스에서 공유될 수 있다.

5. 데이터베이스 설계 기준이 적절히 정의되었는지 확인한다.
- 데이터베이스 설계 표준 지침
ㄴ 데이터베이스 설계 표준 지침 : 테이블, 컬럼, 제약조건을 어떻게 작성할 지 정한 문서
- 데이터 모델 설계 지침의 작성 여부 및 적절성 확인
ㄴ 엔티티, 속성, 관계를 어떻게 표현할지 정한 문서

6. 데이터 모델링이 충분하고 적절하게 수행되었는지 확인
- 논리 데이터 모델의 개념 완정성 및 정확성 확인
- 무결성이 보장이 되었는지 확인
- 엔티티 간 실제 업무 규칙과 맞는지 확인
- 모델 관련 산출물 작성의 적합성 및 일관성에 대해 확인

7. 엔티티 및 프로세스 간의 연관 관계가 명확한지 확인
- 엔티티와 프로세스간 연관관계를 CRUD 매트릭스를 통해 확인

8. 데이터에 대한 접근권한 및 통제가 명확히 분석되었는지 확인
- 데이터 중요도 및 데이터 암호화 가이드라인 확인하기

물리 데이터모델 설계
- 논리모델을 적용하고자 하는 기술에 맞도록 상세화해가는 과정

반정규화
- 정규화를 조금 포기하고, 조회가능성을 높이기 위해 데이터를 중복하거나
테이블 구조를 바꾸는 설계 방법

반정규화 하는 이유
- 데이터가 적을 때는 괜찮지만, 데이터가 많아지면 조회 성능이 느려질 수 있다.
- 그래서 자주 조회하는 데이터는 일부로 합치거나 중복해서 저장한다.

반정규화의 종류
중복테이블 추가하기
- 집계 테이블 추가
- 진행 테이블 추가
- 특정 부분만을 포함하는 테이블 추가

테이블 조합
- 항상 같이 사용되는 두 개 이상의 테이블을 하나로 합치는 것

테이블 분할
- 수직 분할
- 수평 분할

테이블 삭제
- 테이블 재정의나 컬럼의 중복화로 더 이상 사용되지 않는 테이블이 발생할 경우 삭제한다.

컬럼 중복
- 필요한 해당 테이블이나 컬럼을 중복하여 추가한다.
