논리 모델링을 통해 얻은 내용을 데이터베이스에 사용할 수 있는 물리데이터 모델로 변환

엔티티 -> 테이블로 변환
ㄴ일반적으로 테이블과 엔티티 명칭은 동일하게 하는것을 권고
속성 -> 컬럼으로 변환
ㄴ 속성의 명칭과 반드시 일치할 필요는 없으나, 개발자와 사용자 간의 의사소통을 위하여 가능한 표준화된 언어를 사용하도록 한다.
ㄴ SQL 예약어 사용은 피해야 한다.
ㄴ SQL 문장의 가독성을 높이기 위해 컬럼명칭은 가능한 짧은것이 좋다.
UID -> 기본키(PK)로 변환
관계 -> 외래키(FK)로 변환
ㄴ 외래키로 설정된 컬럼의 이름도 보통은 기본키쪽의 이름을 그대로 사용
ㄴ 다른 의미를 가지면 바꿔도 된다
컬럼 타입 및 길이를 정의
ㄴ 내가 넣으려는 데이터의 크기를 파악해서 컬럼의 길이를 지정
반정규화 적용 여부를 검토

물리 데이터 저장소 구성
테이블 제약조건
기본키와 외래키로 관계를 맺고 있는 테이블에서의 제약조건

삭제할 때
- 참조된 기본키의 값이 삭제될 경우 어떻게 처리할지 정하는 규칙
고객이 삭제되면, 그 고객의 주문 데이터는 어떻게 할 것인가

CASCADE
- 부모테이블의 데이터가 삭제되면, 그 데이터를 참조하고 있던 자식 테이블의 데이터도 함께 삭제되는 방식

FOREIGN KEY (customer_id) REFERENCES customer(customer_id)
ON DELETE CASCADE

RESTRICTED
- 자식 테이블에서 사용 중인 값이 있으면 부모 데이터를 삭제하지 못하게 막는 방식

FOREIGN KEY (customer_id) REFERENCES customer(customer_id)
ON DELETE RESTRICT

Nullify
- 부모데이터가 삭제되면, 자식테이블의 외래키 값을 NULL로 바꿔라
- 자식테이블의 외래키 컬럼이 반드시 NULL을 허용해야 한다.

FOREIGN KEY (customer_id) REFERENCES customer(customer_id)
ON DELETE SET NULL

수정할 때
CASCADE
- 기본키 내용이 수정되면 참조하는 외래키 값도 같이 수정되는것

FOREIGN KEY (customer_id) REFERENCES customer(customer_id)
ON UPDATE CASCADE

RESTRICT
- 외래키로 참조되고 있으면 기본키의 값을 수정하지 못하도록 막는다.

FOREIGN KEY (customer_id) REFERENCES customer(customer_id)
ON UPDATE RESTRICT
Nullify
- 부모 테이블의 기본키 값이 수정되면, 자식 테이블의 외래키 값을 NULL로 바꾸는것

FOREIGN KEY (customer_id) REFERENCES customer(customer_id)
ON UPDATE SET NULL

인덱스 설계
- 데이터를 빠르게 찾기 위해 만드는 검색용 목차

인덱스 적용 기준
컬럼의 분포도가 10 ~ 15% 이내인 경우
분포도 : 해당 컬럼의 값이 얼마나 다양하게 나뉘어 있는지 의미
분포도 = (1/컬럼의 도메인 개수) X 100

회원번호 -> 모두가 다르다
10000

분보포 = (1/10000) X 100 = 0.01%

- 이런 컬럼은 인덱스 효과가 크다

10% 이하 : 점점 효율이 좋아진다.
10 ~ 15% : 일반적으로 사용할 수 있다
15% 이상으로 올라갈수록 : 인덱스 효율이 안좋아진다

분포도 범위가 이상이더라도 부분 처리를 목적으로 하는 경우

orders
주문상태	건수
완료		90000
배송중	8000
취소		2000

실제 업무에서 자주 조회하는 데이터가 '취소' 2000건이라면 전체의 2%만 조회하게 된다
- 이런 경우 부분 처리를 목적으로 인덱스를 사용할 수 있다.
- 단순히 값의 종류만 보지말고 실제 조회 조건에서 몇건지 조회되는지 봐야한다.

조회 및 출력 조건으로 사용되는 컬럼인경우
- 입출력 표나 화면 설계서에서 자주 검색 조건으로 사용되는 컬럼인 경우

주문
주문번호
회원ID
주문일자
주문상태

- 누가 언제 주문을 하는지 조회를 많이 한다면 회원ID랑 주문일자에 인덱스를 고려할 수도 있다

기본키랑 UNIQUE 제약조건은 인덱스를 자동으로 생성한다.

인덱스 컬럼의 선정
- 분포도가 좋은 컬럼은 단독 인덱스로 생성한다.
분포도가 좋다는 것은 검색 결과가 적게 나오는 컬럼이라는 뜻

회원번호	이메일
1		a@test.com
2		b@test.com
3		c@test.com
4		d@test.com
5		e@test.com

인덱스는 보통 특정 컬럼에 조건을 걸 때 사용이 된다.

select *
from users
where email = 'c@test.com';

CREATE INDEX idx_user
ON users(email);

결합 인덱스를 사용하는 경우
- 조건이 여러개 걸릴 때 

select *
from orders
where member_id = 10
and order_date >= '2026-01-01';

CREATE INDEX idx_member_id_order_date
on orders(member_id, order_date);

결합 인덱스는 컬럼 순서가 중요하다

수정이 빈번하지 않은 컬럼을 선정한다.
- 인덱스는 조회를 빠르게 해주지만, 삽입,수정,삭제 작업에는 부담이된다.
- 인덱스로 선정한 컬럼의 정리 내용을 따로 저장을 하기 때문에 테이블 데이터가 바뀔 때 인덱스도 같이 바뀌어야 한다.

설계시 고려사항
새 인덱스가 기존 액세스 경로에 영향을 줄 수 있다.
- 액세스 경로 : DBMS가 데이터를 찾기 위해 선택한 이동 방법
대표적인 액세스 경로
1. Full Table Scan(전체 테이블 검색)
- DB가 처음부터 끝까지 전부 읽는다.
2. Index Scan(인덱스 검색)
- 인덱스를 사용해서 필요한 위치만 바로 찾는다.

지나치게 많이 만들면 오버헤드가 된다.
오버헤드 : 빠르게 찾기 위해 만들어둔 목차가 너무 많아서, 오히려 관리가 힘들어지는 것
- 인덱스가 많으면 조회는 빨라질 수 있지만, 데이터 변경 작업은 느려진다.

인덱스는 추가 저장공간이 필요하다.
- 인덱스는 테이블과 별도로 저장되는 검색 구조이다.
- 인덱스를 만들면 저장 공간을 추가로 사용한다.

넓은 범위를 인덱스로 처리하면 오히려 느릴 수 있다.
- 적은 데이터를 빠르게 찾을 때 효과적이다.
- 하지만 전체 데이터의 많은 부분을 조회하면 오히려 전체 테이블을 읽는것이 더 빠를 수 있다.

인덱스와 테이블 데이터의 저장 공간을 적절히 분리한다.
- 대규모 DB에서는 테이블 데이터와 인덱스 데이터를 물리적으로 분리해서 저장하기도 한다.

뷰(View) 설계
- SELECT 한결과를 테이블인것 처럼 저장하는 가상 테이블

뷰가 필요한 이유
복잡한 SQL을 단순하게 사용할 수 있다.
특정 컬럼만 보여줄 수 있다.
조건에 맞는 데이터만 보여줄 수 있다.

뷰 속성
REPLACE
뷰가 이미 존재하는 경우 기존 뷰를 새 내용으로 다시 생성하는 속성
CREATE OR REPLACE 뷰이름 AS
SELECT문

MYSQL & ORACLE 둘 다 됨

FORCE
기본 테이블이 존재하지 않아도 뷰를 생성하는 속성

ORACLE만 됨

- 뷰를 사용하려고 하면 에러가 발생할 수 있다.

NOFORCE
ORACLE에서만 사용 가능
- 참조하려는 테이블이 무조건 존재해야 뷰 생성 가능

WITH CHECK OPTION
- 뷰의 조건을 만족하는 행만 INSERT, UPDATE할 수 있도록 제한하는 속성
- 원본 테이블이 아닌 뷰에다가 INSERT, UPDATE를 하는 것

CREATE VIEW 뷰이름 AS
SELECT...
WITH CHECK OPTION;

WITH READ ONLY
- 뷰를 조회 전용으로 만드는 속성이다.
- INSERT, UPDATE, DELETE는 불가능하다
CREATE VIEW 뷰이름 AS
SELECT...
WITH READ ONLY;

뷰 설계시 고려사항
뷰는 결국 테이블을 조회한다.
- 실제 데이터를 저장하지 않는다.
- 뷰는 내부적으로 SELECT문을 실행해서 결과를 보여준다.
- 그래서 뷰를 사용한다고 해서 무조건 빨라지는건 아니다
- 오히려 뷰 안의 SQL이 복잡하면 성능 문제가 생길 수 있다.

복잡한 뷰는 성능이 느려질 수 있다.

뷰 안의 SELECT문의 조건도 인덱스를 고려해야 한다.

파티션
- 하나의 큰 테이블을 내부적으로 여러 조각으로 나누어 저장하는 기술
- 겉으로는 하나의 테이블로 보여진다.

왜 파티션이 필요한가?
- 대용량 데이터를 여러 영역으로 나누어 저장함으로써, 조회시 필요한 데이터만 읽어 성능을 향상시키고, 백업,복구,삭제 같은 관리 작업을 효율적으로 수행하기 위해 사용한다.

파티션 프루닝

ALTER TABLE orders DROP PARTITION p2024;

해시 분할(Hash Partitioning)
- 특정 컬럼 값을 계산(Hash)해서 데이터를 여러 구역에 균등하게 나누어 저장하는 방법

범위 분할의 경우 -> 사람이 직접 분할의 기준을 정한다.

2024년 -> p2024
2025년 -> p2025
2026년 -> p2026

해시분할은 DB가 직접 계산해서 자동으로 분배를 한다.

데이터 편중 문제
2024년에는 주문이 많았다. -> p2024 파티션에 너무 많은 데이터가 들어감
2025년에는 주문이 적다.

해시 분할을 사용

CREATE TABLE users(
	user_id INT,
	name VARCHAR(50)
)
PARTITION BY HASH(user_id)
PARTITIONS 4;

DB는 내부적으로 해시값 % 파티션 개수 방식으로 계산을 한다.

1 % 4 = 1
2 % 4 = 2
3 % 4 = 3
4 % 4 = 0
5 % 4 = 1
6 % 4 = 2
7 % 4 = 3
8 % 4 = 0

데이터를 추가할 때 특정 파티션에 추가하는게 아니라 DB가 자동으로 계산해서 저장한다.

해시 분할의 장점
데이터 분산 : 특정 파티션에 데이터가 몰리는 현상을 감소시킨다.
저장 균형 : 여러 파티션 크기를 비슷하게 유지
특정 키 조회 빠름 : ID 기본 조회에 적합
대용량 처리 : 부하를 여러 파티션으로 분산할 수 있다.

해시 분할의 단점
범위 조회에 약함 : 날자 범위 검색 비효율
사람이 예측 어렵다 : 어느 파티션에 있는지 알기 어렵다
파티션 개수 변경이 어렵다 : 재분배 작업이 발생할 수 있다.

언제 많이 사용할까?
- 회원 테이블
- 사용자 정보
- 고객 테이블
- 게임 사용자 데이터
- ID 기반 조회가 많은 시스템

조합 분할(Composite Partitioning)
- 파티션 기준을 2개 이상 조합해서 데이터를 나누는 방식
- 먼저 큰 기준으로 한 번 나누고, 그 안에서 다시 작은 기준으로 한 번 더 나누는 방식

조합 분할을 사용하는 이유
- 대용량 데이터 관리 : 1차 파티션만으로 부족할 때 사용
- 조회 범위 축소 : 날짜 조건과 고객 조건을 함께 자주 조회할 때 좋다.
- 데이터 분산 : 특정 연도난 월에 데이터가 올리는 문제를 줄일 수 있다.
- 관리 편의성 : 연도별 삭제, 백업 같은 작업을 하면서 내부 분산도 가능하다.

언제 사용하면 좋을까?
- 주문 테이블 1차 : 주문일 2차 : 고객번호
- 로그 테이블 1차 : 생성일 2차 : 사용자 ID
- 결제 테이블 1차 : 결제일 2차 : 회원 ID
- 게시글 이력 테이블 1차 : 작성일 2차 : 작성자 ID

언제 사용하지 않는게 좋을까?
- 데이터가 많지 않을 때
- 단순 CRUD 중심일 때
- 날짜 조건만으로 충분할 때
- 파티션 관리 경험이 부족할 때
- 조회 조건이 파티션 기준과 맞지 않을 때

파티션 장점
- 읽어야 하는 데이터 범위를 줄여서 성능을 향상
- 전체 데이터 훼손 가능성 감소 및 데이터 가용성 향상
- 각 분할 영역을 독립적으로 백업하고 복구가 가능하다.
- 디스크 스트리핑으로 I/O성능 향상

디스크 스트리핑 : 데이터를 여러 디스크에 나눠서 동시에 저장하거나 읽는 방식

파티셔닝 순서
- 파티션 종류 결정(범위,분할,조합)
ㄴ 데이터를 어떤 방식으로 나눌지 결정
범위 분할 : 값의 범위로 나눈다
해시 분할 : 해시 계산 결과로 나눈다
조합 분할 : 두 가지 방식을 함께 사용
- 파티션을 할 키를 설정
ㄴ데이터를 나눌 기준이 되는 컬럼을 결정
ㄴ 좋은 파티션 키는 데이터를 어느 정도 고르게 나누거나 자주 조회되는 범위를 효과적으로 줄여줄 수 있어야 한다.
ㄴ 이력 데이터는 생성주기 또는 소멸주기와 맞춘다.
ㄴ 전체 테이블에서 오래된 데이터를 한 건씩 삭제하는 것보다 파티션 단위로 관리하는것이 효율적이다.

- 파티션의 개수를 결정
데이터의 양 : 한 파티션이 너무 커지지 않게 한다
조회 패턴 : 자주 조회하는 단위와 맞춘다.
보관 정책 : 삭제,백업 단위와 맞춤
관리 편의성 : 너무 많은 파티션은 피한다.
I/O분산 : 특정 파티션에 너무 몰리지 않게 한다.

클러스터 설계
- 자주 조회되는 데이터를 디스크의 가까운 위체 모아 저장해서 조회 속도를 높이는 물리적 저장방법이다.

일반 저장 방식은 책이 아무 책장이나 흩어져 있는 상태
클러스터 저장 방식은 자주 같이 보는 책들은 같은 책작이나 가까운칸에 모아두는 방식
그렇기 때문에 조회할 때는 바르다
하지만 책을 새로 넣거나 위치를 바꾸거나 삭제할 때는 정리 비용이 더 소모된다.

파티션 : 저장공간을 나눈다.
클러스터 : 관련 데이터를 가까이 저장한다.

클러스터 설계 방법
Oracle 기준

CREATE CLUSTER customer_order_cluster(
	customer_id NUMBER
)

customer_id가 같은 데이터끼리 같은 공간에 저장할 창고를 만들어라

클러스터 인덱스를 만든다
CREATE INDEX idx_customer_order_cluster
ON CLUSTER customer_order_cluster;

클러스터를 사용하는 테이블을 생성
CREATE TABLE customer(
	customer_id NUMBER PRIMARY KEY,
	customer_name VARCHAR2(50) NOT NULL,
)
CLUSTER customer_order_cluser(customer_id);

CREATE TABLE orders(
	order_id,
	customer_id
	...
)
cluster customer_order_cluster(customer_id);

같은 customer_id를 가진 데이터가 물리적으로 가까운 위치에 저장될 수 있다.

MySQL
- 오라클 처럼 CREATE CLUSTER를 통해 클러스터를 생성하지는 않는다.
- InnoDB 테이블의 PRIMARY KEY가 자동으로 클러스터형 인덱스가 된다.
- MySQL 공식 문서에서도 InnoDB는 테이블에 primary key를 정의하면 그것을 clustered index로 사용한다고 적혀있다.

InnoDB
- MySQL에서 테이블 데이터를 저장하고 관리하는 저장 엔진이다.

PK를 만들면 곧 클러스터가 된다.

적용 기준
분포도가 넓을수록 유리한 기법
- 일반적으로 인덱스는 조건으로 조회할 때 데이터가 적을수록 유리하다.
- 그런데 조회범위가 너무 넓으면 인덱스를 타도 많은 행을 계속 찾아가야 해서 오히려 부담이 된다.

SELECT *
FROM orders
WHERE order_date BETWEEN '2026-01-01' AND '2026-03-31'

- 3개월치 주문 데이터가 매우 많다면 인덱스로 하나하나 찾아가는 비용이 커진다.
- 이런 경우 관련 데이터를 물리적으로 가까이 모아두는 클러스터가 도움이 된다.

클러스터는 액세스 기법이 아니라 물리적 저장방법이다.
SELECT *
FROM ORDERS
WHERE customer_id = 10;
- 클러스터는 sql문법에서 직접 "클러스터로 검색해라"라고 지시하는 개념이 아니다.
- 클러스터는 미리 데이터를 저장할 때 관련있는 데이터를 가깝게 저장한다.

분포도가 넓은 테이블은 저장 공간 절약이 가능하다
- 클러스터링을 잘하면 같은 기준의 데이터가 모여 저장된다.
customer_id = 1 인 주문들
customer_id = 2 인 주문들
customer_id = 3 인 주문들
- 이렇게 같은 키 값을 가진 데이터들이 모이면, 반복되는 키 관리나 블록 접근 측면에서 저장 효율이 좋아질 수 있다.
- 단, 무조건 저장 공간이 줄어드는 것은 아니다
- 설계가 잘못되면 오히려 공간 낭비가 생길 수 있다.

작은 테이블
-> 그냥 전체 조회해도 빠름
-> 클러스터 설계 효과가 작다

큰테이블
-> 여러 블록에 데이터가 흩어질 수 있다.
-> 클러스터 설계효과가 커질 수 있다.

- 클러스터는 데이터가 충분히 많고, 조회 패턴이 반복적으로 발생할 때 의미가 있다.

인덱스를 사용한 처리가 부담되는 넓은 분포도에 활용한다
- 조회할 데이터가 너무 많으면 인덱스에 반복접근이 많아져서 부담된다.
- 클러스터는 관련 데이터를 물리적으로 모아두므로, 넓은 범위를 읽을 때 블록 접근 수를 줄이는데 도움이 된다.

여러 테이블이 빈번히 조인될 때 사용

정규화 때문에 분할된 반복 컬럼에 활용한다.

설계 고려사항
- 검색은 빨라지지만 입력,수정,삭제는 느려질 수 있다.
- 새로운 데이터를 넣을 때 어디에 넣어야 같은 그룹끼리 모일까를 고려해야 한다.
- DB는 단순히 빈 공간에 넣는 것이 아니라, 클러스터 구조를 유지해야 한다.

UNION, DISTINCT, ORDER BY, GROUP BY가 빈번한 컬럼으로 고려한다.
- 이 연산들은 데이터를 묶거나 정렬하거나 중복제거하는 작업이다.

SELECT department_id, COUNT(*)
FROM employees
GROUP BY department_id;

department_id를 기준으로 데이터가 물리적으로 모여있다면 그룹화 작업이 유리할 수 있다.

select *
from orders
order by order_date;

order_date 기준으로 데이터가 어느정도 모여있으면 정렬 부담이 줄어들 수 있다.
 
하지만 무조건 적용되는 것은 아니다
해당 컬럼으로 조회, 정렬, 그룹화가 자주 발생하는지 확인해야 한다.

수정이 자주 발생하지 않는 컬럼이 좋다.
- 클러스터 기준 컬럼은 자주 바뀌지 않는 것이 좋다.
- 회원 id -> 거의 바뀔일 없음
반면 status는 자주 바뀔 수 있다
결제 대기 -> 결제 완료 -> 배송중 -> 배송완료
