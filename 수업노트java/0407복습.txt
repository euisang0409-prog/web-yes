리플렉션(reflection)
- 클래스의 정보(필드,메서드, 생성자 등)를 프로그램 실행 중에 조회하고 조작하는 기술

Class 클래스
왜 사용하는가?
1. 객체를 동적으로 생성할 수 있다.

		  컴파일환경			     런타임환경
자바코드 -> 컴파일 -> 바이트코드(.class) -> 실행

일반적인 객체의 생성
Car car = new Car(); -> 컴파일 시점에 클래스가 확정

Class clazz = Class.forName("클래스명") -> 실행 중에 클래스 결정이 가능

무슨 도움이 되는가
- 플로그인 구조 : 기본 프로그램에 외부 기능을 끼워넣는 것

메인프로그램 : 이미지 편집기
외부플러그인 : 흑백 필터, 블러 필터, 선명하게 필터

2. 보통의 프로그램은 미리 정해진 클래스만 사용한다.
하지만 어떤 경우에는 실행하기 전까지 어떤 클래스를 쓸지 모르는 경우가 있다.

- 프로그램에 새로운 기능을 나중에 추가하고 싶다.
- 설정파일에 따라 다른 객체를 만들고 싶다.
- 외부 개발자가 만든 기능을 꽂아 넣듯이 연결하고 싶다.

3. 프레임워크에서 자동 처리
- 나중에 우리가 직접 안써도 스프링부트 내부에서 많이 사용중이다.

싱글톤: 객체를 1개만 생성해서 메모리를 효율적으로 사용 필요할때마다 객체를
만들게 되면 메모리를 계속 차지하게 됨

로그(log) : 프로그램이 실행되면서 남기는 기록
- 사용자가 로그인을 시도 
- 주문내역을 저장할 때
- 에러가 발생했을 때

Class 클래스의 역할
- 클래스의 정보를 알아와야 할 때
- 생성자를 읽어서 객체를 만드는것(프로구램이 실행되는 시점)

핵심
컴파일 시점에 확인하는 것이 아니라 실행시점에서 확인하기 위함이다.

어노테이션
코드에 붙이는 표식같은 것
클래스, 필드, 메서드 등에 붙여서 코드에 대한 추가 정보를 전달할 때 사용합니다.

코드 자체의 실행문은 아니지만 이 코드가 어떤 의미인지, 컴파일러가 무엇을 확인해야 하는지,
프레임워크가 어떻게 처리해야 하는지와 같은 정보를 알려주는 역할을 한다.

어노테이션을 왜 사용할까?

1. 실수를 줄여준다.

2. 코드의 의미를 더 분명하게 보여준다.

자바가 제공하는 어노테이션
@Override
@Dreprecated : 더 이상 업데이트 x  
@SupressWarnings : 경고 보기 싫을 때 사용

메타  어노테이션
@Retention : 언제까지 정보를 유지할거냐
@Target : 어디에 적용할거냐

나만의 어노테이션
public @interface 어노테이션명{}

제네릭 타입을 미리 고정하지 않고 나중에 정할 수 있게 만드는 문법

어떤 타입이든 담을 수 있는 틀을 만든 뒤 사용할 때 타입을 정하는 방식

제네릭을 사용하게 되면 
꺼낼때 형변환이 필요없고, 잘못된 타입을 넣으면 컴파일 오류가 난다.
타입마다 변수를 만들어줄 필요가 없어서 재사용이 크게 증가한다.
제네릭 타입변수에 들어갈 수 있는것은 클래스만 가능하다.

제네릭  메서드
메서드에만 적용되는 제네릭 타입을 따로 두는 것
제네릭클래스가 아니어도 제네릭 메서드를 만들 수 있다.
public <U> void print(U value){
	System.out.println(value);
}
